# Разговорный ИИ-помощник REPORTIX

Помощник ведёт диалог с предпринимателем (голосом или текстом) и выполняет
операции учёта. Первая версия умеет **приёмку/создание товара**:

> — Прими 20 синих футболок размера L по 45 тысяч, продажа 79 тысяч
> — На склад «Основной»?
> — Да
> — Создать товар «Футболка» и оприходовать 20 шт на «Основной» (закуп 45 000, продажа 79 000)?
> — Подтверждаю
> — Готово: создан товар «Футболка», оприходовано 20 шт на «Основной».

## Главный принцип безопасности

**ИИ только понимает и предлагает — он никогда не пишет в базу.**

1. Вход (текст или голос → STT) поступает «мозгу» помощника.
2. Мозг извлекает намерение и поля (что за товар, цена, количество) и/или
   трактует ответы «да/нет/подтверждаю».
3. Детерминированный движок ведёт состояние диалога, дозапрашивает недостающее,
   показывает итог и **ждёт явного подтверждения**.
4. Реальная запись происходит **только после «Подтверждаю»** и **только** через
   уже существующие атомарные сервисы (`quick_create_product` и т.п.) с проверкой
   прав. ИИ не имеет прямого доступа к БД.

Так галлюцинация или ошибка распознавания не может ничего испортить — человек
подтверждает конкретный черновик.

## Контракт API

`POST /api/assistant/message/`

Тело (JSON или multipart):

| Поле | Тип | Обязательно | Описание |
|---|---|---|---|
| `conversation_id` | строка | нет | id диалога; если пусто — начнётся новый |
| `text` | строка | да\* | сообщение пользователя |
| `audio` | файл | да\* | голос (альтернатива text; прогоняется через STT) |
| `language` | строка | нет | язык распознавания (по умолчанию `uz-UZ`) |

\* нужно `text` **или** `audio`.

Ответ:

```json
{
  "conversation_id": "b3f1...",
  "reply": "На склад «Основной»?",
  "state": "collecting",
  "draft": null,
  "result": null
}
```

- `state` ∈ `collecting` | `confirm` | `done` | `cancelled`.
- `draft` заполняется на шаге `confirm` (что именно будет создано + короткоживущий
  `token`), чтобы «Подтверждаю» относилось только к последнему предложению.
- `result` заполняется на шаге `done` (созданный товар).

Состояние диалога хранится на сервере в кэше (Redis в проде, locmem в тестах),
ключ — по тенанту, пользователю и `conversation_id`, TTL 30 минут. Клиенту
достаточно возвращать `conversation_id`.

## Сменный «мозг» (как у распознавания речи)

Мозг выбирается настройкой `ASSISTANT_BRAIN` — доменный код от неё не зависит:

| Значение | Что это | Когда |
|---|---|---|
| `rule` (по умолчанию) | Детерминированные правила + существующий NLU | Работает **без ключей и без интернета**; на нём идут все тесты |
| `gemini` | Gemini через Google Cloud Vertex AI | Один счёт GCP, **без анкет Model Garden и гео-ограничений** — рабочий вариант для Узбекистана |
| `claude_vertex` | Claude через Google Cloud Vertex AI | Требует одобрения Model Garden (Anthropic); для UZ страна недоступна в форме |
| `claude` | Claude по прямому ключу Anthropic | Отдельный ключ `ANTHROPIC_API_KEY` |

Claude-мозг при любой ошибке (нет доступа, плохой ответ) **откатывается на
правила** — помощник продолжает работать. Модель по умолчанию —
`claude-opus-4-8` (`ASSISTANT_MODEL`).

Настройки:

```
ASSISTANT_BRAIN=rule            # rule | claude_vertex | claude
ASSISTANT_MODEL=claude-opus-4-8
VERTEX_PROJECT_ID=...           # для claude_vertex (+ ADC на сервере)
VERTEX_REGION=us-east5
ANTHROPIC_API_KEY=...           # для claude
```

## Права

Приёмка/создание товара — только `owner`/`manager` (как `ManageCatalog`).
Точка API закрыта этим правом, плюс движок повторно проверяет роль перед
выполнением (защита в глубину). Кассир к приёмке не допускается.

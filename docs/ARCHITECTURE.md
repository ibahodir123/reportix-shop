# REPORTIX — система учёта розничного магазина

> Розничный POS-продукт бренда REPORTIX. Система управления розничной торговлей:
> товары, склад, касса (POS), закупки, клиенты, отчёты. Мультифилиальность.
> Веб-платформа (клиентская касса + админ-панель владельца).
>
> **Зафиксированные решения (2026-07-12):**
> - Модель поставки: **SaaS, мультитенантность с самого начала** (много клиентов).
> - **Вариации товара (размер/цвет) — обязательны** в MVP.
> - **Голосовой ввод** наименований и цифр товаров — ключевая фича (см. §9).

---

## 1. Что это и для кого

Облачный SaaS для розничных магазинов (одежда, обувь, электроника, продукты,
аптеки и т.д.). Три роли пользователей:

| Роль | Что делает | Интерфейс |
|------|-----------|-----------|
| **Владелец / администратор** | Настройка, аналитика, справочники, филиалы | Админ-панель |
| **Менеджер / товаровед** | Закупки, приёмка, инвентаризация, ценообразование | Админ-панель |
| **Кассир / продавец** | Продажи, возвраты, приём оплаты | Касса (POS) |

## 2. Основные модули (домен)

### 2.1 Каталог товаров
- Товар (Product): наименование, SKU, штрихкод(ы), категория, бренд, ед. изм.
- **Вариации** (размер/цвет) — критично для одежды/обуви. Product → Variant.
- Категории (дерево), бренды, единицы измерения.
- Цены: закупочная, розничная, оптовая; наценка; история цен.
- Изображения, атрибуты.

### 2.2 Склад и остатки
- Склады/локации (у каждого филиала свой склад).
- Остатки по варианту × складу (Stock).
- Движения товара (StockMovement): приход, расход, списание, перемещение,
  корректировка. **Любое изменение остатка — только через движение** (аудит).
- Приёмка (GoodsReceipt) от поставщика.
- Перемещения между складами (Transfer).
- Инвентаризация (StockCount) с расхождениями и корректировкой.

### 2.3 Продажи / Касса (POS)
- Смена кассира (CashierShift): открытие/закрытие, X/Z-отчёт, инкассация.
- Чек продажи (Sale) + позиции (SaleItem).
- Оплата: наличные, карта, смешанная, в долг; сдача.
- Скидки: на позицию и на чек; промо-акции.
- Возвраты (Return) с привязкой к чеку.
- Быстрый поиск товара по штрихкоду/названию.

### 2.4 Закупки
- Поставщики (Supplier).
- Заказ поставщику (PurchaseOrder) → Приёмка (GoodsReceipt).
- Взаиморасчёты с поставщиками (оплаты, долги).

### 2.5 Клиенты и лояльность
- Клиент (Customer): телефон, история покупок.
- Бонусы/скидочные карты, накопительная скидка.
- Продажа «в долг» и погашение.

### 2.6 Организация и доступ
- Организация (тенант) → Филиалы (Branch) → Склады/Кассы.
- Пользователи, роли, права (RBAC).
- Мультитенантность: данные изолированы по организации.

### 2.7 Отчёты и аналитика
- Продажи (по дню/филиалу/кассиру/товару).
- Прибыль/маржа (по себестоимости FIFO или средневзвешенной).
- Остатки и стоимость склада.
- Топ товаров, ABC-анализ, оборачиваемость.
- Отчёт по смене (Z-отчёт).

## 3. Технологический стек (рекомендация)

Совпадает с твоим текущим опытом (Django+React+Docker) — быстрый старт и
переиспользование наработок.

| Слой | Технология | Почему |
|------|-----------|--------|
| Backend | **Python 3.12 + Django 5 + Django REST Framework** | Знаком, быстрый CRUD, admin из коробки, транзакции |
| БД | **PostgreSQL 16** | Надёжность, оконные функции для отчётов, JSONB |
| Кэш/очереди | **Redis + Celery** | Фоновые задачи (отчёты, экспорт), кэш остатков |
| Frontend | **React 18 + TypeScript + Vite** | SPA, знаком |
| UI-kit | **Ant Design** (или Mantine) | Готовые таблицы/формы для админки, ускоряет |
| Стейт/данные | **TanStack Query** + Zustand | Кэш серверных данных, простой клиентский стейт |
| Аутентификация | JWT (SimpleJWT) или сессии | — |
| Инфра | **Docker Compose** | dev/prod паритет |
| Прод | Nginx + Gunicorn | Как в текущем проекте |

**POS-фронтенд** — отдельное приложение (или маршрут) с упором на скорость,
работу со сканером штрихкодов (клавиатурный ввод) и крупные кнопки.

> Альтернатива, если хочешь мобильную кассу позже: PWA (offline-first через
> IndexedDB + очередь синхронизации). Заложить в архитектуру, но не в MVP.

## 4. Ключевые архитектурные решения

1. **Остатки — производная от движений.** Таблица `Stock` кэширует текущий
   остаток, но источник истины — `StockMovement`. Каждая продажа/приёмка/
   списание создаёт движение в транзакции. Это даёт полный аудит и
   восстановимость.

2. **Себестоимость.** На MVP — средневзвешенная (проще). FIFO-партии — фаза 2.
   Маржа считается на момент продажи и фиксируется в `SaleItem.cost_price`.

3. **Мультитенантность.** Поле `organization_id` на всех бизнес-сущностях +
   middleware, ограничивающий queryset текущей организацией. Изоляция на уровне
   приложения (проще, чем schema-per-tenant, достаточно для старта).

4. **Идемпотентность продаж.** POS шлёт `client_uuid` на каждый чек, чтобы
   повтор при сбое сети не задваивал продажу.

5. **Числа/деньги.** Все суммы — `Decimal`, количество — `Decimal` (весовой
   товар). Никаких float.

6. **Штрихкоды.** У товара может быть несколько штрихкодов; поиск — по индексу.

## 5. Черновик доменной модели (ядро)

```
Organization ─┬─ Branch ─┬─ Warehouse ── Stock ── (Variant)
              │          └─ CashRegister ── CashierShift ── Sale ── SaleItem
              ├─ User (role, branch)
              ├─ Category (tree)
              ├─ Brand
              ├─ Product ── Variant (size/color, barcode, SKU)
              ├─ Supplier ── PurchaseOrder ── GoodsReceipt ── StockMovement
              ├─ Customer
              └─ StockMovement (type: in/out/writeoff/transfer/adjust)
```

Продажа (упрощённо):
```
Sale(id, organization, branch, register, shift, customer?, cashier,
     number, client_uuid, subtotal, discount, total, paid, change,
     payment_type, status, created_at)
SaleItem(sale, variant, qty, price, discount, cost_price, total)
```

## 7. Дорожная карта (по фазам)

### MVP (фаза 0) — «магазин может продавать и видеть остаток»
1. Организация + филиал + пользователи/роли (минимум).
2. Каталог: товары + вариации + категории + штрихкоды.
3. Склад: приёмка → остатки → движения.
4. Касса: смена, продажа, оплата (наличные/карта), чек.
5. Базовые отчёты: продажи за день, остатки.

### Фаза 1 — «полноценный учёт»
- Возвраты, скидки/промо, продажа в долг.
- Закупки: заказы поставщику, взаиморасчёты.
- Инвентаризация, перемещения между складами.
- Клиенты + лояльность.
- Отчёт по марже/прибыли, Z-отчёт смены.

### Фаза 2 — «масштаб и интеграции»
- FIFO-себестоимость, партии, сроки годности.
- Фискализация / онлайн-касса (по требованиям РУз).
- Интеграции: платёжные (Payme/Click), маркетплейсы, Telegram-уведомления.
- Offline-режим кассы (PWA + синхронизация).
- Мобильное приложение (по необходимости).

## 9. Голосовой ввод товаров (ключевая фича)

Цель: предприниматель диктует голосом наименование и цифры (цена, количество,
штрихкод), система заполняет карточку товара или строку прихода. Работает для
**узбекского и русского** языка (в РУз часто смешанная речь).

### Сценарии
- **Голосовое заведение товара:** «Футболка синяя, размер эль, закуп 45 тысяч,
  продажа 79 тысяч, 20 штук» → черновик карточки с полями.
- **Голосовая приёмка:** быстрый ввод «наименование — количество — цена» подряд.
- **Голосовой поиск на кассе:** назвать товар вместо набора.

### Архитектурный подход (по слоям)
1. **Захват аудио** — в браузере через `MediaRecorder` (getUserMedia). Кнопка
   «микрофон» в форме товара/прихода. Опция потоковой записи для длинных серий.
2. **Speech-to-Text (STT).** Два пути, закладываем абстракцию `STTProvider`:
   - *Облачный* (точнее для uz/ru): внешний STT-сервис с поддержкой узбекского.
     Отправляем аудио → получаем текст. Быстрый старт.
   - *On-prem* (приватность/оффлайн, фаза 2): self-hosted распознавание.
   Провайдер выбирается конфигом, доменный код от него не зависит.
3. **Парсинг сущностей (NLU) — самое важное.** Из текста выделяем структуру:
   `{ name, attributes(size/color), purchase_price, sale_price, qty, unit }`.
   - Слой нормализации чисел: «сорок пять тысяч» / «qirq besh ming» → `45000`.
     Отдельный модуль `number_words` для ru **и** uz (словами и цифрами).
   - Извлечение атрибутов по словарю (цвета, размеры S/M/L/XL, «эль», «эм»).
   - Сопоставление наименования с существующим товаром (fuzzy match), иначе —
     предложить создать новый.
4. **Подтверждение пользователем.** STT/NLU **никогда не пишут в БД напрямую** —
   всегда возвращают заполненную форму-черновик, человек проверяет и сохраняет.
   Это защищает от ошибок распознавания цен/количеств.

### Решения
- Голосовой слой — **отдельное приложение** `voice/` в backend: принимает аудио,
  вызывает STT, гоняет NLU, отдаёт структурированный черновик. Изолирован, чтобы
  менять провайдера STT без затрагивания домена.
- Числа и деньги из голоса проходят ту же валидацию, что и ручной ввод (Decimal,
  границы). Никакого доверия к сырому распознаванию.
- Начинаем с формы «товар/приёмка», кассовый голосовой поиск — фаза 1.

## 8. Дорожная карта — уточнения по решениям

- Мультитенантность (SaaS): `organization_id` + middleware c самого начала, план
  тарифов/подписок — фаза 2.
- Вариации обязательны: доменное ядро строится вокруг `Variant`, остаток и
  продажа всегда на уровне варианта (у «плоского» товара — один дефолт-вариант).
- Голосовой ввод (§9) входит в MVP-контур каталога/приёмки.
